Deals POC · Architecture Brief

Belief Stack for financial reasoning integrity

Status: Early-stage research

A proposed architecture for runtime integrity monitoring of multi-step LLM financial reasoning traces. The objective is detecting and intervening when reasoning departs from evidentiary integrity — not general-purpose AI governance, not a content classifier, not a replacement for human review.

Domain: deals / accounting / valuation / reconciliation workflows.
Architectural pattern: Belief Stack (L0 evidence → L1 operational regions → L2 priors → L3 lifecycle → L4 calibration → governance eligibility).
Companion: generate_synthetic_data.py, run_demo.py, run_demo_multipass.py in this directory.

SECTION 1Problem definition

Modern LLM systems can execute sophisticated multi-step financial reasoning workflows — accounting normalization, transaction modeling, reconciliation, valuation calculations, debt and covenant logic, portfolio aggregation, waterfall structures, scenario analysis.

These systems often produce outputs that appear coherent and authoritative while silently introducing:

  • unsupported assumptions
  • arithmetic drift
  • conditional-logic failures
  • source/evidence mismatches
  • sign inversions
  • cross-table mapping corruption
  • reconciliation inconsistencies

Traditional evaluation methods assess final-answer correctness, point-in-time classifier outputs, or static policy violations. They generally do not model the step-by-step reasoning trajectory, evidence integrity across steps, runtime drift from source data, operational failure patterns, or intervention opportunities during execution.

The proposed architecture treats the reasoning process itself as a dynamic execution trace that can be monitored, modeled, calibrated, and intervened upon.

SECTION 2Core concept — evidence substrate vs reasoning trace

The system operates on two primary artifacts:

A. Evidence substrate (L0)

Structured source data — JSON financial records, ledger entries, accounting tables, validation outputs, source mappings, transaction metadata, reconciliation rules, tie-out constraints. This layer is the authoritative evidence base. It is not interpreted; it is the raw evidence field.

B. Reasoning trace

The model-generated execution path — markdown reasoning traces, intermediate calculations, mapping decisions, formula applications, corrections, validation commentary, reconciliation steps, final conclusions. This layer represents the model's evolving reasoning process.

The central architectural objective

Detect when the reasoning trace departs from evidentiary integrity — and intervene during execution, before downstream corruption compounds.

The hypothesis: different reasoning operations exhibit distinct historical failure modes and therefore require distinct runtime safeguards. The a priori choice is justified by the domain — operations like FORMULA_APPLICATION or RECONCILIATION have hard, named invariants (operands resolve to evidence, totals tie out) that justify defining the regions ahead of time rather than discovering them.

Worth validating empirically

The eleven candidate regions are a starting typology. Phase 1 manual audit should confirm whether real traces partition cleanly into these operations, or whether the typology needs splitting, merging, or extension. If a substantial fraction of reasoning steps don't fit, the typology is wrong — and the discovery itself is signal.

L3 — Lifecycle & trajectory

L3 models how the reasoning process evolves over time: where drift begins, whether corrections self-heal downstream calculations, whether unsupported assumptions propagate, whether reconciliation pressure forces revision, whether repeated correction loops emerge.

Candidate trajectory states:

  • stable — operations passing invariants step-by-step
  • drifting — invariants beginning to weaken
  • unstable — multiple invariants failing in sequence
  • corrected — a CORRECTION operation has resolved a prior failure
  • unreconciled — a RECONCILIATION operation has failed
  • escalated — runtime halt triggered; human review required

The trajectory itself becomes an object of analysis — not just "did the final answer fail?" but "at which step did integrity break, and how did the trace respond?"

Trajectory state requires warrant composition, not label sequence

A sequence of three operation labels — say, EVIDENCE_EXTRACTION → SIGN_NORMALIZATION → CONDITIONAL_APPLICATION — is just three labels. It doesn't tell you whether the trace is stable, drifting, or corrected. The trajectory state emerges from how the warrants compose:

  • Did each step's evidence_refs resolve cleanly into the next step's inputs? → stable
  • Did a step introduce an operand whose evidence_ref doesn't validate? → drifting
  • Did a downstream CORRECTION re-establish a clean warrant chain? → corrected
  • Did a RECONCILIATION operation fail to balance dependent quantities? → unreconciled

L3 is not over the labels — it's over the warrant chain. That's why the L1 contract (label + warrant) is load-bearing for everything above it. Without warrants, L3 has no signal to compose; with warrants, the trajectory state falls out of warrant validity propagation step-by-step.

L4 — Calibration & continuous evaluation

L4 evaluates which interventions reduce failure rates, which operational regions are most fragile, which evidence structures correlate with drift, whether priors remain valid under changing workflows, and whether interventions still outperform baseline. This layer continuously pushes back against stale assumptions — including the typology choices in L1.

SECTION 4L1 representation contract (label + warrant)

This brief inherits the same L1 representation contract used in the other Belief Stack applications: every L1 assignment must carry both a label and a warrant. In this domain, a reasoning step assigned to FORMULA_APPLICATION is not enough — the assignment must include the evidence supporting it.

{
  "step_id": "s14",
  "operation_type": "FORMULA_APPLICATION",
  "warrant": {
    "inputs":              ["EBITDA", "Revenue"],
    "evidence_refs":       ["json.company.ebitda", "json.company.revenue"],
    "validation_status":   "PASS",
    "risk_flags":          ["derived_operand"],
    "confidence":          0.84,
    "downstream_deps":     ["valuation_step_19"]
  }
}
Operating disciplines (inherited from the broader Belief Stack)
  1. Do not use a learned (or a priori) structure unless its applicability is measured.
  2. Do not emit priors outside coverage. If a reasoning step doesn't fit any operational region cleanly, return UNCLASSIFIED — don't force-assign.
  3. Do not treat region assignment as truth. The label is a claim about what kind of operation this is; the warrant is the evidence supporting the claim.

The third discipline is particularly load-bearing here. A reasoning step might syntactically look like a formula application but reference operands that don't exist in evidence — that's not a passing FORMULA_APPLICATION, it's an assignment with an empty warrant, which should be treated as a near-failure state, not as a normal operation.

Label and warrant must be emitted together

The hardest implementation question this raises: at runtime, who assigns the operation_type to a raw line of reasoning text?

One tempting answer is "a downstream text classifier inspects each step and labels it." This is a known failure mode: a post-hoc classifier with no warrant of its own emits confident labels for content it doesn't actually understand — markdown table boundaries, section dividers, or formatting wrappers can be confidently mislabeled as FORMULA_APPLICATION or RECONCILIATION when there's nothing operational happening in that span at all.

The principle is not "structural emission instead of post-hoc classification." The principle is:

Simultaneity discipline

Whoever assigns the operation_type must emit the warrant in the same act. Label without warrant is broken regardless of when it runs. Label with warrant (operand resolution, evidence_refs, validation_status) is fine, post-hoc or not — because the warrant either holds up against the evidence substrate or it doesn't.

Several mechanisms satisfy this discipline:

  • FSM-constrained generation — force the model to declare its operation_type and emit its evidence_refs before printing any calculation
  • Structured output schemas (JSON-mode, function-calling) — require the warrant fields to be filled before the call completes
  • Post-hoc classifier with validation — fine if the classifier's verdict includes the warrant fields and the validation_status reflects whether the warrant actually verifies against evidence

The failure mode to avoid: a classifier that emits a label but cannot supply a warrant — or worse, supplies a warrant that doesn't actually validate but is reported as PASS. A label without a verified warrant is not signal; it is confident noise.

L0 cleaning is a substrate-specific preprocessing discipline

Before any L1 assignment, the reasoning trace's container noise must be stripped from the substantive content. The traces are markdown — tables, headers, dividers, list markers (|, ---, ##, ↳) need to be parsed away so the L1 step sees the propositional content and formulas, not the formatting characters. This is a preprocessing step that precedes everything else; the rest of the brief assumes the input has already been through it.

SECTION 5Initial experimental approach

Phase 1 — Manual audit

Start with 5–10 traces. For each trace:

  1. Segment markdown into discrete reasoning steps
  2. Identify operation type for each step
  3. Map referenced values back to JSON evidence
  4. Validate arithmetic where applicable
  5. Detect unsupported assumptions
  6. Detect conditional-logic mismatches
  7. Detect reconciliation failures
  8. Identify where downstream corruption begins

Goal: understand the anatomy of reasoning failure before building generalized infrastructure.

Phase 2 — Reasoning ledger

Construct a normalized execution ledger using the schema above. Goal: create structured observability over reasoning behavior; turn each trace into an inspectable, replayable object.

Phase 3 — Failure pattern analysis

Analyze common operation-level failures, recurring correction patterns, reconciliation pressure points, unsupported-assumption propagation, drift signatures, high-risk operation sequences. Goal: discover which operational regions deserve runtime intervention — and which don't, because their failure rate is low or their failures don't propagate.

Phase 4 — Runtime intervention prototypes

Build candidate interventions for the highest-yield regions identified in Phase 3:

  • Formula enforcement — emit explicit formula, bind operands to evidence, validate arithmetic before continuing
  • Conditional enforcement — quote governing condition, evaluate condition state, prohibit unsupported branch execution
  • Reconciliation enforcement — verify totals, verify downstream consistency, halt on unresolved mismatch

Phase 5 — Earned governance

Interventions are not deployed by declaration. Each must define a measurable target, demonstrate improvement vs baseline, prove transferability across traces, and remain calibration-aware over time.

What "measurable improvement" means is not the same bar for every intervention. The substrate-aware principle that governed L1 also governs governance:

Intervention classRight barWhy
Hard-invariant enforcement
(formula, reconciliation, sign normalization)		 Binary certification
0% structural breakage across test fixtures		 The invariant is deterministic. There is no statistical version of "the totals tied out 80% of the time." Tolerance is exactly zero.
Soft-judgment enforcement
(assumption flagging, evidence quality scoring, classification confidence)		 Statistical lift
improvement vs baseline above noise threshold		 The judgment is probabilistic. Some false positives are acceptable; the bar is whether the intervention shifts the rate enough to be worth the inference cost.
Escalation / runtime consultation Two-sided cost-benefit
downstream error reduction net of review load		 Escalation has a real cost (human review time). The bar isn't "does it catch errors" alone — it's "does it catch errors at an acceptable false-escalation rate."

Same earned-governance discipline applies to every intervention type; the bar just looks different depending on what the intervention is trying to do.

SECTION 6Architectural hypothesis

Central hypothesis

Multi-step financial reasoning systems fail in structured, repeatable operational patterns.

If the hypothesis holds:

  • Reasoning integrity can be modeled
  • Failure risk can be localized to specific operations
  • Interventions can be scoped to the operations that need them
  • Runtime behavior can be adaptively constrained
  • Escalation can be triggered before downstream corruption compounds

The architecture does not attempt to model financial truth globally. It attempts to track when reasoning departs from evidentiary integrity. That is a much narrower, more tractable claim — and the right scope for a first deployment.

SECTION 7Constraints & unknowns

This work is exploratory. Open questions include:

  • Whether operational regions remain stable across workflows (or whether each new workflow needs its own typology)
  • Whether interventions generalize across domains (deals vs accounting vs valuation vs portfolio aggregation)
  • Whether priors drift under changing models or prompts
  • How much structure is intrinsic to the operations vs corpus-specific
  • Whether lightweight runtime interventions outperform retraining approaches
  • Whether trajectory modeling materially improves downstream integrity, or whether per-step invariants are sufficient on their own
  • Whether the eleven a priori regions partition real traces cleanly, or whether the typology needs reshaping
What this is — and is not
  • This is runtime integrity infrastructure research.
  • This is not a proven governance solution.
  • This is not a replacement for human review.
  • This is not a guarantee of correctness — it's a discipline for noticing when reasoning has departed from evidence.

SECTION 8Long-term direction

If the architecture proves useful in financial reasoning workflows, plausible extensions include:

  • Online calibration (priors that update as new traces accumulate)
  • Intervention lifecycle management (interventions earn / lose deployment eligibility over time)
  • Reasoning replay systems (rebuild a trace from its ledger to debug failure modes)
  • Invariant-aware execution protocols (operations refuse to run without their warrants)
  • Trajectory-aware escalation (halt patterns recognized from prior failures)
  • Adaptive runtime consultation layers (model asks for clarification when invariants are weak)
  • Cross-domain operational reasoning maps (transfer of operation typologies between deals / accounting / valuation)

The initial focus, however, remains deliberately narrow:

Scope for the first deployment

Improve integrity and observability in multi-step financial reasoning workflows. Detect when the reasoning trace departs from evidentiary integrity. Intervene where intervention is shown to help. Earn eligibility through measured improvement, not declaration.
