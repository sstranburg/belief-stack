# Synthetic themes ā€” beliefstack demo

A minimal end-to-end walk through the five layers on synthetic internal
MS Teams chatter. The point of this example is to make the L2 *stateful
expectation* and the L3 *belief revision* visible ā€” not to model any
particular organization.

> **Data is not shipped.** The L0 snippets are LLM-generated at run time and the
> demo needs `OPENAI_API_KEY` set (used for both generation and embeddings).
> Run `python generate_synthetic_data.py` then `python run_demo.py` /
> `python run_demo_multipass.py` to produce the data, outputs, and reports
> locally. This example is the second domain (alongside `deals_thesis_radar`)
> exercising the same L2 `extras` abstraction; the generator stays example-local
> by design and is not promoted into the core library.

## What this demonstrates

| Layer | In this demo |
|---|---|
| L0 events | ~80 LLM-generated short Teams snippets across 8 organizational themes |
| L1 regions | K-means clusters of the snippets (`k=8`) over OpenAI embeddings |
| L2 expectation | LLM-generated **stateful expectation** per region: `narrative` + `state` (ESCALATING / RESOLVING / STALLING / MIXED) + `score` (0ā€“100) + `direction` + `conviction` + `read` + `predicted_class` |
| L3 lifecycle | First-pass `born` events; multi-pass transitions (born ā†’ strengthened / weakened / contradicted / retired) driven by `predicted_class` accuracy |
| L4 calibration | Walk-forward 70/30 split; per-region top-1 / top-3 / Brier on `predicted_class` vs held-out outcomes |
| Decisions | PROMOTE / MONITOR / RECLUSTER per region |

## L2 is the object; `predicted_class` is the bridge

The richer expectation (narrative / state / score / read) is the L2 object
the demo produces. `predicted_class` is a discrete label tucked into the
expectation so the existing L3 / L4 / decisions machinery can score it ā€” it
is the **evaluation bridge**, not the belief itself. Reports lead with the
narrative + state, not the predicted class.

The expectation lives in `Hypothesis.extras` ā€” a forward-compatible payload
channel added to the core library so domain-specific generators can carry
richer L2 content without modifying the base type. The library does not
interpret `extras`; consumers (the report renderer, inspection surfaces,
downstream callers) do.

## Substrate

Synthetic internal MS Teams chatter at a mid-sized tech company. Eight
cross-functional themes:

- `customer_escalation_pattern`
- `vendor_renewal_friction`
- `platform_migration_concerns`
- `budget_pressure_signals`
- `staffing_capacity_pressure`
- `compliance_audit_findings`
- `team_restructure_signals`
- `product_launch_readiness`

Outcome label space (carried on each event for L4 scoring):

- `escalated` ā€” theme moved up in priority / drew formal attention
- `resolved` ā€” theme was addressed and closed out
- `stalled` ā€” theme went dormant without resolution

Each theme's train/test outcome pattern is deliberately mixed:

- **stable** themes ā€” calibrate cleanly ā†’ PROMOTE
- **shifting** themes ā€” modal class changes between train and test ā†’ STRENGTHENED / WEAKENED / INVERTED
- **noisy** themes ā€” no clear modal outcome ā†’ MONITOR / RECLUSTER

That variety is the point: a real region-card report should not look uniform.

## Environment requirements

The demo is the LLM-driven path. It fails fast if the key is missing.

```bash
export OPENAI_API_KEY=sk-...

# optional overrides
export OPENAI_MODEL=gpt-4o-mini             # chat model for L2 + data gen
export OPENAI_EMBED_MODEL=text-embedding-3-small   # embedding model
```

```bash
pip install -e ".[openai]"
```

Approximate cost per full multi-pass run (data gen + embeddings + L2 calls
across two passes) is on the order of **$0.05ā€“0.20** with `gpt-4o-mini`.

If you want the deterministic (mock-embedder + empirical-generator) path,
look at the library tests under `tests/` ā€” the demo deliberately does not
take that path.

## Data

`data/synthetic_events.jsonl` is generated by:

```bash
python examples/synthetic_themes/generate_synthetic_data.py
```

The generator issues 8 LLM calls (one per theme, each returning 10 snippets)
and caches the result. Re-run only when you want fresh chatter.

Every event is tagged `SYNTHETIC: ā€¦` in its text and
`"source": "synthetic_generator"` in metadata. No real entities appear.

## Run

```bash
pip install -e ".[openai]"
python examples/synthetic_themes/generate_synthetic_data.py
python examples/synthetic_themes/run_demo.py
open examples/synthetic_themes/outputs/region_cards.html
```

## Multi-pass lifecycle (the belief-revision view)

`run_demo.py` runs one calibration pass ā€” every region's lifecycle is `born`.
That proves the pipeline works but doesn't show the *belief revision*.

`run_demo_multipass.py` splits the same data into three chronological eras
and runs two calibration passes:

```
era 1   Feb 1 - Mar 2     baseline      expectations born
era 2   Mar 3 - Apr 1     first revision  born -> strengthened / weakened
era 3   Apr 2 - Apr 29    second revision    -> contradicted / promoted / retired
```

Run it:

```bash
python examples/synthetic_themes/run_demo_multipass.py
open examples/synthetic_themes/outputs/region_cards_multipass.html
```

Each card shows:

1. The **expectation** at the top: narrative, state pill, score bar, read.
2. The **calibration stats** below: top-1 / top-3 / Brier on `predicted_class`.
3. The **lifecycle transition**: `born ā†’ strengthened`, with the top-1 delta
   that drove it (e.g. `top1 0.50 ā†’ 0.75`).

The expectation reads as the object. The lifecycle transition is the belief
revision the architecture is built to surface.

### Threshold note

The multi-pass demo uses lower lifecycle thresholds than the library defaults
(`min_n_active=2`, `delta=0.10`) because this synthetic dataset is tiny ā€”
~3 events per region per era. Production deployments stay at the defaults
(`min_n_active=5`, `delta=0.05`).

## Caveats

- The synthetic distributions are illustrative, not realistic.
- 80 events across 8 regions is tiny. Production-grade clustering needs
  orders of magnitude more.
- The `LLMTeamsExpectationGenerator` is example-local ā€” it lives next to
  this example, not in the core library. Promotion into the library is
  gated on at least two substrates exercising the same abstraction.
- The LLM expectation can be wrong, miscalibrated, or generic ā€” the
  lifecycle layer's job is to surface that over time, not to prevent it.
