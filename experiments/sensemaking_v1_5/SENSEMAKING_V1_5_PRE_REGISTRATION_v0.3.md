# Sensemaking v1.5 — Pre-registration (v0.3)

Derivative of [SENSEMAKING_V1_5_PRE_REGISTRATION_v0.2.md](SENSEMAKING_V1_5_PRE_REGISTRATION_v0.2.md). v0.1 and v0.2 results stand unchanged; v0.3 is reported separately with a three-way head-to-head. Locked 2026-05-30.

**v0.3 scope** — narrow but architecturally substantive:

1. **Promote lifecycle revision-prediction to the primary measurement.** L3 carried directional signal in v0.2 that L2 state buckets lost under the corrected baseline. v0.3 makes that finding the headline.
2. **Demote static state buckets to secondary.** They remain reported, head-to-head with v0.1 and v0.2.
3. **REPRICING becomes its own primary bucket within the state-bucket measurement** — no longer folded into Constructive / Cautious by direction (v0.2's choice), no longer in Ambiguous (v0.1's choice).
4. **EARLY remains its own standalone bucket** (unchanged from v0.2).
5. **Add per-sub-window stratification** to test the Cautious failure mode against the rising-tide hypothesis.

Same window, same horizons, same lookahead, same labeling protocol, same exclusions.

---

## 0. Why this document exists

v0.2 surfaced two architectural findings:

- **Static state buckets are brittle to baseline composition.** v0.1's apparent Constructive 20D advantage of +2.97% dissolved under v0.2's tightened baseline.
- **Lifecycle revision events preserved directional signal.** Under the v0.2 baseline, Constructive_revision − Cautious_revision = +0.79% at 5D (right direction), while static-state Constructive − Cautious = −1.48% (wrong direction).

The v0.2 report's §9 named both as v0.3 candidates. This pre-registration locks the v0.3 response.

---

## 1. Success criterion

The v0.3 measurement tests whether **lifecycle revision events** produce separable forward outcome distributions at 5D and 20D horizons under rolling walk-forward evaluation. The report publishes per-bucket and per-horizon metrics for the lifecycle primary, head-to-head against v0.1 and v0.2; static state buckets are reported as a secondary axis.

Static state bucket head-to-head also tests whether REPRICING-as-standalone produces a more interpretable bucket structure than v0.1 (REPRICING-in-Ambiguous) or v0.2 (REPRICING split by actor direction).

Per §1.1 of v0.1 (preserved): calibrated reporting, not pass/fail. No pre-committed significance threshold.

---

## 2. Universe

Unchanged from v0.1 / v0.2. Same window (2025-12-05 → 2026-05-26), same primary universe (31 tickers from `backtest_history.parquet` baseline variant, excluding experimentals USAR / MP / ODC per §2.2), same I-001 / I-002 limitations.

Lifecycle event universe: drawn from `data/derived/expectation_lifecycle_events.parquet`, restricted to events whose `event_type` is a revision-class event (reconfirmed, strengthened, contradicted, weakened) and whose joined ticker is in the primary universe. Born / retired events remain excluded from the revision-prediction measurement (they are field-population events, not revision events).

## 3. Lifecycle event buckets (PRIMARY in v0.3)

### 3.1 Event-type to bucket mapping

Unchanged from v0.1 / v0.2 §11.1 definition:

| Bucket | Event types |
|---|---|
| **Constructive_revision** | reconfirmed, strengthened |
| **Cautious_revision** | contradicted, weakened |

Restated here because it is the v0.3 primary measurement.

### 3.2 Evaluation date semantics

For each lifecycle event in scope:

- **T** = the lifecycle event date.
- Forward 5D and 20D relative returns are computed from T using the per-ticker price series, identical to the state-bucket measurement (per §4 below, unchanged from v0.1).
- The same §7 exclusion rules apply: T+5 and T+20 must fall within the window, price data must be present.

### 3.3 Limitation noted in pre-reg

Lifecycle events were generated by the v1 sensemaking pipeline using LLM-judged conviction transitions. The v0.3 measurement does not re-derive lifecycle events; it reads them as the v1 pipeline emitted them, exactly as v0.1 and v0.2 §11.1 did. Any biases in the LLM-judge layer carry forward.

## 8. Control / baseline

Both axes use the v0.2 Ambiguous baseline definition: rows in MACRO, PRICE-LED, UNCLEAR from the state-bucket measurement. This is a deliberate choice — it lets lifecycle metrics be compared to state-bucket metrics on a common baseline, so the head-to-head can show whether lifecycle separates from baseline better than state buckets do.

REPRICING is **not** in the baseline (consistent with v0.2). Per the v0.2 measurement, REPRICING_bullish averages near-zero forward returns and would re-introduce the v0.1 dilution if folded back into Ambiguous.

## 10. What v0.3 does NOT claim

Unchanged in spirit from v0.1 / v0.2. Additionally:

- v0.3 does NOT claim lifecycle revision-prediction is live-predictive. The v0.2 internal-gap finding (Constructive_rev − Cautious_rev > 0 at 5D under tightened baseline) is what v0.3 promotes; v0.3 measures whether that internal gap holds up as the primary axis.
- v0.3 does NOT claim REPRICING-as-standalone is the correct bucket assignment. It claims standalone reporting is more interpretable than the v0.1 (in Ambiguous) and v0.2 (split by direction) approaches.
- v0.3 does NOT claim the sub-window cut is optimally calibrated. The cut is deliberately simple (calendar midpoint) so the discipline is auditable.

## 11. Secondary measurements

### 11.1 Static state buckets (demoted from primary)

| Bucket | States (v0.3) | Change from v0.2 |
|---|---|---|
| Constructive | CONFIRMED, DISAGREEMENT | REPRICING_bullish removed |
| Cautious | NEG_CONFIRMATION, DIVERGENCE | REPRICING_bearish removed |
| **REPRICING_primary** | REPRICING (all rows, no direction split) | NEW standalone bucket |
| Early_followthrough | EARLY | unchanged from v0.2 |
| Ambiguous | MACRO, PRICE-LED, UNCLEAR | unchanged from v0.2 |

The state-bucket measurement reuses v0.2's labeling protocol (Constructive: fwd_rel > 0 = Hit; Cautious: fwd_rel < 0 = Hit; Early_followthrough: Constructive labeling, provisional). **REPRICING_primary is unlabeled** for Hit/FP/FN because v0.3 makes no directional commitment about REPRICING; its distribution is reported descriptively (n, avg, median, % positive, baseline diff).

### 11.2 Per-sub-window stratification

The v0.3 pre-registration commits to **two equal-calendar-day sub-windows**:

- **Sub-window A (early):** 2025-12-05 → 2026-02-28 (87 calendar days, ~58 trading days)
- **Sub-window B (later):** 2026-03-01 → 2026-05-26 (87 calendar days, ~60 trading days)

Cut date 2026-03-01 chosen as calendar midpoint of the 173-day window. The cut is intentionally simple — it does not attempt to align with market-regime boundaries. v0.3 reports per-sub-window numbers; if a regime-aware cut is needed, that is a v0.4 candidate.

Per-sub-window measurement runs for:

- **All five state buckets** (Constructive, Cautious, REPRICING_primary, Early_followthrough, Ambiguous)
- **Both horizons** (5D, 20D)

The Cautious bucket is the focal interest (per the rising-tide hypothesis in v0.2 §9), but all buckets get the sub-window split so any window-structural effect is visible.

### 11.3 Warrant coverage (preserved)

Same as v0.2 §11.2. Partition the primary universe by `sufficient_data ∈ {True, False}`. Re-run state-bucket metrics per partition. No change in protocol.

### 11.4 Lifecycle warrant coverage (NEW supplementary)

For lifecycle events, partition by whether the event's joined state-bucket row has `sufficient_data = True` at the same (date, ticker). Reports per-partition lifecycle bucket metrics. Tests whether lifecycle revisions on warrant-sufficient rows separate more cleanly than on warrant-insufficient rows.

## 12. Sensitivity appendix (v0.3)

Restructured to match v0.3's promoted primary.

### 12.1 Lifecycle event-type granularity

Split each lifecycle bucket into its constituent event types:

- Constructive_revision → reconfirmed (n) vs strengthened (n) — separate metrics
- Cautious_revision → contradicted (n) vs weakened (n) — separate metrics

Tests whether the bucket-level signal is driven by one event type or distributed across both. If reconfirmed dominates strengthened (or vice versa), that is a v0.4 candidate for splitting the bucket.

### 12.2 Sub-window robustness for lifecycle primary

Re-run the §3.1 lifecycle measurement per sub-window from §11.2. Tests whether the lifecycle directional signal holds in both early and later sub-windows or is concentrated in one.

### 12.3 Experimental tickers included (preserved from v0.1 / v0.2)

Include USAR / MP / ODC (only MP present in `backtest_history.parquet` per I-002). v0.3 bucket mappings otherwise.

### 12.4 Sensitivity is not the primary claim (preserved)

Same constraint as v0.1 / v0.2.

## 16. What v0.3 will say publicly

If lifecycle revision-prediction holds up as primary:

> v0.3 promoted lifecycle revision events to the primary forward-calibration measurement. The L3 directional gap surfaced in v0.2 (Constructive_revision &gt; Cautious_revision at 5D) survives as the headline; static state buckets remain reported as a comparison surface. The L3 layer carries forward-outcome information that L2 buckets do not, on this substrate.

If it does not:

> v0.3 tested whether lifecycle revision events could carry the primary measurement axis. They did not. The internal directional gap observed in v0.2 did not survive promotion to primary scrutiny. The next iteration's scope shrinks to a more local question about why lifecycle events appeared informative under the v0.2 baseline but not under v0.3's full primary measurement.

Either outcome is a v0.3 result. The discipline is what makes either one publishable.
